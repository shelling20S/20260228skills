# Impl Plan - 実装計画書作成スキル

設計ドキュメントをもとに、フェーズ分割された実装計画書を作成する。

## 核となる設計思想

1. **担当者の明示**: すべての作業項目に 👤（人間）/ 🤖（AI）を付与する
2. **進捗の可視化**: チェックボックス・状態カラム（✅ / 🚧 / ⏳）で進捗を追跡する
3. **人間タスクの詳細化**: 人間が実施する項目は、1ステップごとの具体的な手順を記載する
4. **外部サービスの手順**: Google Cloud / Supabase / Vercel 等の設定手順はスクリーンショット不要なレベルで詳細に書く

## リファレンススキル（外部サービス手順）

外部サービスの設定手順は、以下のリファレンススキルに分離されている。
実装計画書に組み込む際は、該当するリファレンスの手順を参照し、プレースホルダをプロジェクト固有の値に置換する。

| スキル名 | 内容 | 主なリファレンスID |
|---------|------|-------------------|
| `ref-supabase` | プロジェクト作成、API Keys、MCP接続、Auth設定 | REF-SB-01〜06 |
| `ref-google-cloud` | OAuth同意画面、クライアントID、API有効化、スコープ | REF-GC-01〜06 |
| `ref-vercel-deploy` | ダッシュボードデプロイ、環境変数、再デプロイ | REF-VC-01〜04 |
| `ref-gemini-api` | APIキー取得、環境変数設定、動作確認 | REF-GM-01〜04 |

### 必須読み込みルール（今回追加）

- `prod-impl-plan` を使用した実装計画作成では、**開始直後に必ず** 次の2スキルを読み込む。
  - `/Users/kiyogon/Desktop/skills/ref-google-cloud/SKILL.md`
  - `/Users/kiyogon/Desktop/skills/ref-supabase/SKILL.md`
- 2スキルの参照後、Google Cloud / Supabase の👤手順を計画書へ反映する。
- どちらかのスキルが見つからない場合は、理由を明記してから代替手順で続行する。

### 使い方

実装計画書内で外部サービスの👤手順が必要な箇所に、以下の形式で参照する:

```markdown
### X.X Supabase プロジェクト作成 ⏳

> 参照: ref-supabase REF-SB-01〜03
> プレースホルダ置換: {{プロジェクト名}} → `myapp`, {{リージョン}} → `Tokyo`

| # | タスク | 実施者 | 完了条件 | 状態 |
|---|--------|--------|---------|------|
| X.X.1 | プロジェクト作成 + API Keys取得 | 👤 | 【SB1】〜【SB3】メモ | ⏳ |
| X.X.2 | .env.local 作成 | 👤 | 環境変数設定完了 | ⏳ |
```

各リファレンスの詳細手順をそのまま実装計画に転記するか、参照のみとするかはプロジェクトの規模に応じて判断する。

## 作成プロセス

### Step 1: 情報収集

まず、以下2つのリファレンススキルを**必ず先に**読み込む:
- `/Users/kiyogon/Desktop/skills/ref-google-cloud/SKILL.md`
- `/Users/kiyogon/Desktop/skills/ref-supabase/SKILL.md`

**Supabase MCPを使う計画の場合は、以下を前提チェックとして明記する**:
1. `codex mcp list` で `supabase` が `enabled` である
2. 未接続時は `ref-supabase REF-SB-03` の復旧手順（`remove` → `add` → `login` → IDE再読み込み）を実施
3. 疎通確認は `mcp__supabase__list_projects()` で行う

そのうえで、以下のドキュメントを読み込む:
- `docs/01_spec/requirements.md` → 実装すべき機能要件を把握
- `docs/02_design/basic_design.md` → アーキテクチャと画面構成を把握
- `docs/02_design/technical_architecture.md` → 技術スタックと外部サービスを把握
- `docs/05_test/test_plan.md` → テスト方針を把握

### Step 2: フェーズ分割

以下の基準でフェーズを分割する:

1. **Phase 0（準備ゾーン）**: 必ず最初に配置
   - 外部サービスのセットアップ（Supabase、Vercel、認証等）
   - 開発環境構築
   - **人間タスクが多い** → 詳細手順を必ず記載

2. **Phase 1〜N-1（機能実装）**: 機能ごとに分割
   - 独立してデプロイ可能な単位で分ける
   - 各フェーズに E2E テストチェックリストを含める
   - **AIタスクが多い** → テスト対象を明示

3. **Phase N（リリース準備）**: 必ず最後に配置
   - 本番環境デプロイ
   - 外部サービスの本番設定
   - **人間タスクが多い** → 詳細手順を必ず記載

### Step 3: 担当者の割り振り

| 作業カテゴリ | 担当 | 理由 |
|-------------|------|------|
| 外部サービスのアカウント作成・設定 | 👤 | 認証情報が必要 |
| APIキー取得・環境変数設定 | 👤 | 機密情報の取り扱い |
| コーディング（実装・テスト） | 🤖 | 自動化可能 |
| マイグレーション実行 | 🤖 | MCP経由で実行可能 |
| デプロイ設定（Vercel等） | 👤 | ダッシュボード操作 |
| E2Eテスト実行 | 🤖+👤 | AIが実行、人間が結果確認 |
| 本番動作確認 | 👤 | 実機での確認が必要 |

### Step 4: 人間タスクの詳細手順作成

👤タスクには以下を必ず含める:

1. **アクセスURL**: 具体的なURLを記載
2. **ステップバイステップ**: 「1. ... 2. ... 3. ...」の番号付き手順
3. **入力値**: 具体的な入力値やテンプレート
4. **メモすべき情報**: 後続ステップで使う値を【記号+番号】で明示
5. **トラブルシューティング**: よくあるエラーと対処法

#### 外部サービス手順の書き方

Google Cloud / Supabase / Vercel 等の手順は、公式ドキュメントの手順を参考にしつつ、プロジェクト固有の値を埋め込む:

```markdown

**Step 1: [大項目]**
1. [公式URL] にアクセス
2. 「[UI上のボタン/メニュー名]」をクリック
3. [入力フィールド名]: `[プロジェクト固有の値]`
4. 「[確定ボタン名]」をクリック

> **注意**: [注意事項があれば記載]

**Step 2: [大項目]**
1. [手順...]
2. 以下の値を設定:
   ```
   KEY=value
   ```
3. 「[保存ボタン名]」をクリック

**メモすべき情報**:
- 【X1】[項目名]: `[値の説明]`
- 【X2】[項目名]: `[値の説明]`

**トラブルシューティング**:
| 症状 | 原因 | 対処 |
|------|------|------|
| [エラーメッセージ] | [原因] | [対処法] |
```

### Step 5: 進捗管理の記号

状態カラムには以下の記号を使用:

| 記号 | 状態 | 使い方 |
|------|------|--------|
| ⏳ 未着手 | まだ開始していない | 初期状態 |
| 🚧 進行中 | 作業中 | 着手後に更新 |
| ✅ 完了 | 完了 | 完了条件を満たした後 |
| ⏳ (スキップ) | 意図的にスキップ | オプション項目の場合 |
| ⏳ (オプション) | 実施するか未定 | 必須でない項目 |

Phase 見出しにも状態を付与:
```markdown
## Phase 0: 準備ゾーン ✅ 完了
## Phase 1: 基盤構築（MVP） 🚧 進行中 (60%)
## Phase 2: メモとナレッジ ⏳ 未着手
```

### Step 6: レビューと確認

作成した計画書をユーザーに提示し、以下を確認:
- フェーズ分割は適切か
- 担当者の割り振りは正しいか
- 人間タスクの手順は十分詳細か
- 漏れている作業はないか

## 重要なルール

- **Phase 0（準備）と最終Phase（リリース）は必ず作成する**
- **実装計画の作成時は、`ref-google-cloud` と `ref-supabase` のSKILL.mdを必ず読み込んでから着手する**
- **👤タスクは1ステップごとの手順を省略しない**
- **外部サービスの手順はURL・ボタン名・入力値を具体的に書く**
- **各フェーズにE2Eテストチェックリストを含める**
- **「メモすべき情報」で後続ステップとの依存関係を明示する**
- **進捗更新時は状態カラムとPhase見出しの両方を更新する**
- テンプレートの構造は参考であり、プロジェクトの特性に合わせて柔軟に調整する
