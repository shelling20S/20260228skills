# 設計原則

## 引き算のデザイン

**良いデザインは引き算。一人のデザイナーの目線で。**

### なぜこの原則が重要か

複数のAIや参考サイトから「良いパーツ」を集めて統合すると、一見完成度が上がったように見えるが、実際には：

| 問題 | 症状 | 結果 |
|-----|------|------|
| 足し算の罠 | 各ソースの良い部分を全て入れる | 要素過多で焦点がぼやける |
| 一貫性の喪失 | 異なるデザイン言語の混在 | ちぐはぐな印象 |
| コンセプトの希薄化 | 機能追加優先 | サービスの本質が見えなくなる |

### 正しいアプローチ

```
❌ Geminiの配色 + Claudeのアニメーション + ChatGPTのレイアウト
   → 良いパーツの寄せ集め = 統一感のないデザイン

✅ 一つのベースを選び、そのビジョンを貫く
   → 一貫したビジョン > 良いパーツの寄せ集め
```

### 実践チェックリスト

- [ ] **ベースを一つ選ぶ**: 複数案がある場合、最も「らしさ」があるものを選択
- [ ] **追加より削除**: 機能を足したくなったら「本当に必要か？」を問う
- [ ] **コンセプトに立ち返る**: 迷ったらサービスの本質（PROTOTYPE.mdの概要セクション）を確認
- [ ] **一人の視点を維持**: 途中で別のデザインを混ぜない

### 判断基準

| 質問 | Yes なら採用 | No なら削除 |
|-----|-------------|-------------|
| この要素はコンセプトを強化するか？ | ✅ | ❌ |
| なくても伝わるか？ | ❌（必要） | ✅（削除） |
| 初見で理解できるか？ | ✅ | ❌（簡素化） |

## プロトタイプ基本方針

**目的: 人間がHTMLファイルを開いて、必要最低限の動作確認ができること。**

### 外部連携
- **外部API/サービス連携禁止**: 実際の通信は一切行わない
- **モックで擬似体験**: 連携しているように見せる
  - ログインボタン → 擬似ログイン画面 → そのままログイン
  - LLM呼び出し → 「LLMモックを呼んでいます」フロート表示 → setTimeout後に擬似レスポンス
- **ローディングを挟む**: setTimeout(300〜800ms)で擬似待機。非同期処理の存在を体感させる

### データ
- **永続化はlocalStorage/IndexedDB**: 外部DBは使わない
- **リスト表示は必ず複数件**: サンプルデータを複数件準備
- **境界値データを含める**: 長いテキスト、空文字、改行あり、日本語と英数字混在
- **データリセット機能**: 「サンプルデータに戻す」ボタンを用意（設定画面やフッター等）

### 認証
- **ログインが必要なシステム**: ログイン画面で複数ユーザーを選択してログインできるようにする

### ナビゲーション・反応
- **全アクションに視覚的反応**: ボタン押下→トースト、保存→「保存しました」等。無反応は禁止
- **全画面に戻る手段**: モーダルの閉じる、画面遷移の戻る。行き止まりを作らない
- **現在地の表示**: ヘッダーにタイトル、タブのアクティブ状態等
- **空状態の表示**: データ0件時の画面を用意（初回起動・全削除後）

### 技術制約
- **ビルドツール不使用**: 素のHTML/CSS/JS
- **外部API不使用**: モックデータで対応
- **データ永続化**: localStorage / IndexedDB
- **画面サイズ**: 375px × 812px（モバイルファースト）
