## 2. 4つの観点

### 前提の検証

「このアイデアが成り立つために、何が真でなければならないか？」

チェック項目:
- 市場の前提: 「顧客は〇〇を求めている」は本当か？
- 技術の前提: 「〇〇は技術的に可能」は本当か？
- 行動の前提: 「顧客は〇〇の行動をとる」は本当か？
- 競合の前提: 「競合は〇〇しない」は本当か？

質問例:
```
「〇〇という前提がありますが、これはどのように検証しましたか？」
「もし〇〇でなかった場合、このアイデアはどうなりますか？」
```

### リスクの洗い出し

「最悪のシナリオは何か？」

リスクカテゴリ:
- **市場リスク**: 需要がない、タイミングが悪い
- **技術リスク**: 作れない、スケールしない
- **実行リスク**: チームが足りない、資金が足りない
- **競合リスク**: 真似される、先を越される
- **規制リスク**: 法的問題、コンプライアンス

質問例:
```
「このサービスが失敗するとしたら、最も可能性が高い理由は？」
「競合がこのアイデアを知ったら、何をしてきますか？」
```

### 反証の探索

「このアイデアが間違っている証拠は？」

反証パターン:
- 類似サービスの失敗事例
- 顧客インタビューでの否定的反応
- 市場データとの矛盾
- 技術的な制約

質問例:
```
「同じようなサービスが過去に失敗した事例はありますか？」
「このアイデアに反対する人は、何と言うでしょうか？」
```

### 依存関係の特定

「このアイデアの成功は、何に依存しているか？」

依存カテゴリ:
- **パートナー依存**: 特定企業との提携が必須
- **プラットフォーム依存**: App Store、AWS等のポリシー
- **人材依存**: 特定スキルを持つ人材の確保
- **タイミング依存**: 法改正、社会変化のタイミング
- **資金依存**: 特定額の調達が必須

質問例:
```
「このビジネスが成り立つために、外部のどの要素が必要ですか？」
「その依存先がNoと言った場合、代替案はありますか？」
```

## 4. 会話例

**ユーザー**: 「AI発注アシスタント」をレビューして

**Claude（レビューモード）**:
```
レビューモードで「AI発注アシスタント」を検証します。

## レビュー結果

### 前提の棚卸し
| 前提 | 確度 | 検証方法 |
|-----|-----|---------|
| 飲食店はPOSデータを持っている | 中 | 店舗ヒアリング |
| 発注業務に時間がかかっている | 高 | 既存調査あり |
| AIの提案を信頼して発注する | 低 | 要検証 |
| 仕入れ先のデータ連携が可能 | 低 | 業界構造調査 |

### リスク一覧
| リスク | 影響度 | 発生確率 | 対策案 |
|-------|-------|---------|-------|
| 予測精度が低く廃棄増加 | 高 | 中 | 人間承認フロー |
| 仕入れ先がAPI非対応 | 高 | 高 | FAX/電話代行 |
| 既存システムとの統合困難 | 中 | 中 | POS連携から開始 |

### 反証・懸念点
- **類似サービスの苦戦**: 〇〇社が撤退した事例あり。理由は「現場が使いこなせない」
- **過剰在庫より欠品を恐れる心理**: 発注量を減らす提案は受け入れられにくい

### 依存関係
- **POS連携**: 主要POSベンダーとの提携が必須。代替策は手入力だが、それでは価値半減
- **天候・イベントデータ**: 予測精度向上には外部データAPI必要

---

**次のステップ提案**:
1. 飲食店オーナー5名にモック画面でヒアリング
2. 主要POS 3社に連携可否を確認
3. 類似サービス撤退企業へのヒアリング（可能なら）

収束モードに進んで他の案と比較しますか？
```
