# Proto Design

要件ヒアリングから仕様書作成までを1フローで実行し、`prototype/PROTOTYPE.md` を作成する。

## フェーズ1: 要件ヒアリング

以下の観点で質問を行う（既に情報があればスキップ）：

1. **目的**: 何を解決したいか、誰のためのツールか
2. **主要機能**: 必須の機能は何か
3. **ターゲット**: 想定ユーザー層、利用シーン
4. **制約**: 技術的制約、デザイン要件
5. **優先順位**: MVP（最小限の製品）の範囲

※ `prototype/IDEATION_RESULT.md` が存在する場合は、その内容を要件の土台として使用し、ヒアリングを省略または短縮して仕様書作成に直行する。

## プロトタイプ基本方針

**目的: 人間がHTMLファイルを開いて、必要最低限の動作確認ができること。**

仕様書作成・実装の両方で以下を遵守する。

### 外部連携

| ルール | 詳細 |
|--------|------|
| **外部API/サービス連携禁止** | 実際の通信は一切行わない |
| **モックで擬似体験** | 連携しているように見せる。例: ログインボタン→擬似ログイン画面→そのままログイン |
| **LLM等の非同期処理** | 明示的に「LLMモックを呼んでいます」等のフロートを表示 |
| **ローディング挟む** | setTimeout(300〜800ms)で擬似待機を入れ、非同期処理の存在を体感させる |

### データ

| ルール | 詳細 |
|--------|------|
| **永続化はlocalStorage/IndexedDB** | 外部DBは使わない |
| **リスト表示は必ず複数件** | データが複数件になる箇所には複数件のサンプルデータを準備 |
| **境界値データを含める** | 長いテキスト（折り返し確認）、空文字、改行あり、日本語と英数字混在 |
| **データリセット機能** | 「サンプルデータに戻す」ボタンを用意。壊れた状態から復帰可能にする |

### 認証

| ルール | 詳細 |
|--------|------|
| **ログインが必要なシステム** | ログイン画面で複数ユーザーを選択してログインできるようにする |

### ナビゲーション・反応

| ルール | 詳細 |
|--------|------|
| **全アクションに視覚的反応** | ボタン押下→トースト、保存→「保存しました」等。無反応は禁止 |
| **全画面に戻る手段** | モーダルの閉じる、画面遷移の戻る。行き止まりを作らない |
| **現在地の表示** | ヘッダーにタイトル、タブのアクティブ状態等 |
| **空状態の表示** | データ0件時の画面を用意（初回起動・全削除後） |

### バグ防止設計ルール（必須）

プロトタイプはスピードが命。デバッグに時間がかからないシンプルな設計にする。

#### 画面遷移の設計

| ルール | 詳細 |
|--------|------|
| **画面の表示/非表示は `display:none`/`display:block` を使う** | `opacity`/`pointer-events` で隠すと、キーボード操作や連打で「見えないボタンが押せる」バグの原因になる |
| **各遷移関数の冒頭にガード節を書く** | 前提条件（現在のフェーズ、必要なデータの存在）を最初にチェックし、不正な状態なら安全にトップへ戻す |
| **ボタンは「使えないとき非表示」にする** | disabled + opacity ではなく、display:none で完全に消す。デバッグしにくい「隠れたボタン」を作らない |

#### 状態管理の設計

| ルール | 詳細 |
|--------|------|
| **state と画面を同期させる** | 画面遷移とstateの更新は必ずセットで行う。片方だけ変更しない |
| **データ参照前にnullチェック** | `state.currentXxx` にアクセスする関数は冒頭で `if (!state.currentXxx) return;` を入れる |
| **タイマーにはフェーズガードを入れる** | `scheduleTimer` のコールバック冒頭で「まだ期待するフェーズにいるか」をチェックする |

#### ボタンとイベントの設計

| ルール | 詳細 |
|--------|------|
| **全 `<button>` に `type="button"` を付ける** | デフォルトの `type="submit"` によるページリロードを防ぐ |
| **連打対策** | ボタン押下後に即座にボタンを非表示/disabled にする。処理完了後に次のボタンを表示 |
| **スキップ処理後は次のフェーズへ自動遷移** | スキップ後にユーザーが迷子にならないよう、明示的に次のフェーズへ進む |

#### 仕様書への記載（必須）

状態遷移セクション（セクション6）に以下を含める:
- **各フェーズの遷移条件**（「○○フェーズにいて、かつ○○データが存在するとき」）
- **異常時の動作**（「データがnullの場合はトップに戻す」）
- **ボタンの表示条件**（「○○が完了したときのみ表示」）

## フェーズ2: 仕様書作成

### 作成時の注意

#### データ準備
- 複数発生するデータは**複数件**準備する
- 改行する可能性があるフィールドは**改行データ**を準備する
- サンプルデータに**境界値**を含める（長文、空文字、改行、日英混在）

#### 仕様の粒度
- 実装者が迷わないレベルまで具体化
- 画面の状態（空/ローディング/エラー）を網羅
- ボタン押下時の挙動を明記
- 外部連携のモック仕様を明記（何を表示し、どう振る舞うか）

## PROTOTYPE.md 構成

テンプレートは [references/prototype_template.md](references/prototype_template.md) を参照。

> **注**: キャラクターが必要なプロジェクトは `util-character-sheet` で別途作成し、`prototype/CHARACTERS.md` に保存する。

## ワークフロー

1. 既存情報の確認（ideationの成果物、ユーザーの発言等）
2. 不足情報があればユーザーに質問
3. `prototype/PROTOTYPE.md` を作成（概要〜技術仕様まで一括）
4. 前バージョンがあれば `prototype/history/PROTOTYPE_v[N].md` に保存
5. `prototype/prototype_status.md` を更新（状態: 仕様書作成完了）
6. 自動で `proto-03-doc-review` を実行

## 次のステップ

PROTOTYPE.md作成後、`proto-03-doc-review` を実行してドキュメントレビューを行う。
