# PROTOTYPE: Multi-Artist Tracker for Music Fans

## 1. 概要

### 1.1 目的
国内の音楽ファンが、複数の推しアーティストの「イベント情報（ライブ／ツアー／展示／ポップアップ等）」と「リリース情報（Album／EP／Single）」を、スマホから1つのダッシュボードで俯瞰できるようにする。  
「情報源がバラバラ」「スプレッドシート管理が面倒」「友人との共有がしづらい」といった課題を解消し、Googleカレンダーへの追加もワンタップで行える体験を提供する。

### 1.2 ターゲット
- 日本国内在住の音楽ファン
- 複数アーティストを継続的に追っている人
- 日常的にスマホを利用し、Googleアカウントを持っている人
- 既にメモ・スプレッドシート・カレンダー等で予定管理をしているが、情報整理に手間を感じている層

### 1.3 主要機能（MVP）
1. Googleアカウント風ログイン（実際は擬似ログイン）
2. アーティスト検索・フォロー管理
3. アーティスト詳細ダッシュボード
   - 今後のイベント一覧（ライブ／展示など種別バッジ付き）
   - リリース一覧（Album／EP／Single）
4. イベント詳細＋「Googleカレンダーに追加」リンク生成（モック）
5. 手入力によるイベント・リリース登録
6. 限定リンクによる共有ページ（閲覧専用、個人的情報は非表示）
7. データリセット（サンプルデータに戻す）

### 1.4 制約・前提
- **デバイス**: スマホ中心（幅 375px〜）に最適化しつつ、PCでも破綻しないレスポンシブレイアウト。
- **言語**: 日本語を主とし、主要な見出し・ボタンには英語サブラベルを併記（例: 「アーティスト / Artist」）。
- **外部連携**: 実際の外部API呼び出しは禁止。
  - Googleログインは「ログイン画面でユーザーを選ぶ擬似ログイン」とする。
  - Googleカレンダー連携は、URL生成のみ行い、新しいタブを開くモックとする。
- **データ永続化**: localStorageのみ使用。外部DBやサーバ通信は行わない。
- **UIテイスト**: Spotify / Apple Music 風のカードレイアウトを基本とし、アルバムアート風プレースホルダーやカラーアクセントを用いる。

## 2. 画面一覧

### G0: ログイン画面（擬似 Google Login）
- 目的: ユーザーを選択して「ログイン状態」を作る。
- 要素:
  - サービスロゴ／名称
  - 「Googleでログイン / Sign in with Google」風ボタン（実際はモック）
  - モーダルもしくは次画面で「ユーザー選択カード」を表示（複数のダミーユーザー）

### G1: ホーム／アーティスト一覧
- 目的: フォロー中アーティストの一覧と、追加・削除を行う。
- 要素:
  - ヘッダー: サービス名、現在ログイン中ユーザー名、設定アイコン
  - セクションA: 「フォロー中のアーティスト」カード一覧
  - セクションB: 「アーティストを追加」検索フォーム

### G2: アーティスト詳細ダッシュボード
- 目的: 特定アーティストの今後のイベントとリリース情報を一覧表示するメイン画面。
- 要素:
  - アーティスト名（日英併記可能）、タグ、公式サイト／SNSリンク
  - タブまたはセクション:
    - 「イベント / Events」セクション: カード形式で今後のイベントを時系列表示
    - 「リリース / Releases」セクション: カード形式でリリース一覧を表示

### G3: イベント詳細
- 目的: 個々のイベント情報を詳細に確認し、Googleカレンダーに追加する。
- 要素:
  - イベントタイトル（日英）
  - イベント種別バッジ（LIVE / TOUR / POPUP / EXHIBITION 等）
  - 日時（開始／終了、終日フラグ対応、JST明示）
  - 会場名・会場住所
  - 関連アーティストリスト
  - 公式URL／チケットURL
  - 「Googleカレンダーに追加 / Add to Google Calendar」ボタン

### G4: 共有ページ（公開ビュー／限定リンク）
- 目的: フォロー中アーティストと今後のイベントを、閲覧専用で第三者に見せる。
- 要素:
  - ユーザー名またはニックネーム
  - フォロー中アーティスト一覧（カード）
  - 今後のイベント一覧（簡易カード、元画面より情報少なめ）
  - 行く／行かない・チケット状況・メモ等は**一切表示しない**

### G5: 設定／データリセット
- 目的: 共有リンク管理と、ローカルデータのリセットを行う。
- 要素:
  - 共有リンクON/OFFトグル、リンクコピー用ボタン
  - 「サンプルデータに戻す」ボタン
  - ログアウトボタン

## 3. ユースケース

### UC01: ログインして推しアーティストをフォローする
1. ユーザーはG0で「Googleでログイン」ボタンを押す。
2. 擬似ローディング後、複数のダミーユーザー一覧が表示される。
3. 自分用ユーザーを1つ選択すると、G1（ホーム）に遷移する。
4. 検索フォームでアーティスト名を入力し、検索結果からカードの「フォロー」ボタンを押す。
5. フォローしたアーティストが「フォロー中」一覧にカードとして追加される。

### UC02: アーティスト詳細からイベントを確認し、Googleカレンダーに追加する
1. G1のフォロー中アーティストカードから1つをタップしてG2に遷移。
2. 「イベント」セクションで今後のイベント一覧を確認。
3. 気になるイベントカードをタップしてG3（イベント詳細）に遷移。
4. 詳細情報を確認後、「Googleカレンダーに追加」ボタンを押す。
5. 擬似ローディングを挟んだ後、新規タブでGoogleカレンダー作成画面（URLモック）が開く。

### UC03: 共有リンクを発行して、友人に予定を見せる
1. G1ヘッダーから設定アイコンを押してG5に遷移。
2. 「共有ページを公開」トグルをONにする。
3. 自動生成された限定URLが表示され、「リンクをコピー」ボタンでコピーできる。
4. 友人がそのURLを開くと、G4デザインの共有ページが表示される。
5. 友人はフォロー中アーティストと今後のイベントを一覧で閲覧できるが、行く／行かないなど個人的情報は見えない。

## 4. データモデル

> 実装言語には依存しない概念モデル。実際の保存は localStorage のJSONで行う。

### 4.1 User
- `id`: string（内部ID）
- `displayName`: string（表示名）
- `emailMock`: string（ダミーメールアドレス、画面には基本表示しない）
- `locale`: "ja" | "en"（現在は"ja"固定でもよい）

### 4.2 Artist
- `id`: string
- `nameJa`: string
- `nameEn`: string（無い場合は `nameJa` を流用してもよい）
- `tags`: string[]（ジャンルなど）
- `officialSiteUrl`: string | null
- `snsLinks`: { type: "X" | "Instagram" | "YouTube" | "Other"; url: string }[]

### 4.3 Event
- `id`: string
- `artistIds`: string[]（複数アーティスト出演イベント対応）
- `titleJa`: string
- `titleEn`: string
- `type`: "LIVE" | "TOUR" | "POPUP" | "EXHIBITION" | "OTHER"
- `startAt`: string（ISO形式の日時文字列、JST前提）
- `endAt`: string | null
- `allDay`: boolean
- `venueName`: string
- `venueAddress`: string
- `officialUrl`: string | null
- `ticketUrl`: string | null
- `source`: "sample" | "manual" | "autoMock"
- `userNotesPrivate`: string（ユーザー個別メモ、共有ページには出さない）
- `goingFlag`: "none" | "interested" | "going"（共有ページには出さない）
- `ticketStatus`: "none" | "applied" | "won" | "lost"（共有ページには出さない）

### 4.4 Release
- `id`: string
- `artistId`: string
- `titleJa`: string
- `titleEn`: string
- `type`: "ALBUM" | "EP" | "SINGLE" | "OTHER"
- `releaseDate`: string（YYYY-MM-DD）
- `officialUrl`: string | null

### 4.5 ShareProfile
- `id`: string
- `ownerUserId`: string
- `isPublic`: boolean（trueの場合のみ共有ページ有効）
- `createdAt`: string
- `updatedAt`: string

### 4.6 AppState（実行時状態）
- `currentUserId`: string | null
- `currentView`: "login" | "home" | "artistDetail" | "eventDetail" | "settings" | "shareView"
- `currentArtistId`: string | null
- `currentEventId`: string | null
- `isLoading`: boolean（画面全体のローディング）
- `pendingAction`: "login" | "openCalendar" | null（ローディング中のアクション種別）

### 4.7 永続化（localStorage キー）
- `mat_users`: User[]
- `mat_artists`: Artist[]
- `mat_events`: Event[]
- `mat_releases`: Release[]
- `mat_shareProfiles`: ShareProfile[]
- `mat_lastLoginUserId`: string | null

## 5. 画面仕様詳細

### 5.1 G0: ログイン画面
- レイアウト:
  - 上部: ロゴとサービス名（小さめ）
  - 中央: 「Googleでログイン / Sign in with Google」風ボタン（1つ）
- 挙動:
  - ボタン押下時:
    - すぐにボタンを `display: none` にし、「ログイン中... / Signing in...」ローディングインジケータを表示。
    - setTimeout(500〜800ms) 後に、ユーザー選択UI（小さなモーダル or 新しい画面状態）を表示。
  - ユーザー選択カード押下:
    - `state.currentUserId` をセット。
    - `mat_lastLoginUserId` をlocalStorageに保存。
    - G1（ホーム）に遷移。
- エラー／異常時:
  - ユーザーが1人も存在しない場合は、サンプルユーザーを自動生成してから再表示。

### 5.2 G1: ホーム／アーティスト一覧
- レイアウト:
  - ヘッダー:
    - 左: サービス名
    - 右: ユーザーアイコン（タップで設定画面へ）
  - メイン:
    - セクション「フォロー中 / Following」: アーティストカードが横スクロールまたは縦リストで並ぶ。
    - セクション「アーティストを追加 / Add Artist」: 検索入力＋結果カード。
- アーティストカード:
  - アートワーク風プレースホルダー（円形 or 角丸四角）
  - アーティスト名（日→英）
  - 「詳細を見る / View」ボタン（G2へ）
  - 「フォロー解除 / Unfollow」ボタン
- 挙動:
  - 検索入力後、「検索 / Search」ボタン押下で擬似検索を行い、数件の候補カードを表示。
  - フォローすると `mat_artists` にアーティストが保存され、ユーザーのフォロー一覧に紐づく（MVPでは全ユーザー共通でもよいが、概念上はユーザー別）。
- 空状態:
  - フォロー中0件の場合、「まだフォローしているアーティストはいません」メッセージ＋「アーティストを追加してみましょう」ガイドを表示。

### 5.3 G2: アーティスト詳細ダッシュボード
- レイアウト:
  - ヘッダー: 戻るボタン（G1へ）、アーティスト名（日英）、フォロー状態トグル。
  - 上部: アーティストのカード（大きめ）、公式サイト・SNSアイコン。
  - 下部: タブ風レイアウト
    - タブ「イベント / Events」
    - タブ「リリース / Releases」
- イベントセクション:
  - 今後のイベントをカードでリスト表示（最低3件以上のサンプル）。
  - カード内容:
    - 種別バッジ（LIVE/TOUR/POPUPなど、色付き）
    - タイトル（日英）
    - 日付・時間（JST表記）
    - 会場名（1行）、住所（2行目小さめ）
  - カードタップでG3（イベント詳細）へ。
- リリースセクション:
  - グリッドまたは縦リストでカード表示。
  - カード内容:
    - ジャケット風プレースホルダー
    - タイトル（日英）
    - 種別バッジ（ALBUM/EP/SINGLE）
    - 発売日
    - 公式リンクボタン（押下で新規タブを開くモック）

### 5.4 G3: イベント詳細
- レイアウト:
  - ヘッダー: 戻るボタン（G2へ）、イベントタイトル（短縮表示）。
  - メインセクション:
    - タイトル（日英）
    - タグ（LIVE/TOUR/POPUP/EXHIBITION）
    - 日時（開始日時、終了日時、終日の場合は「終日 / All day」表記）
    - 会場名・住所（タップでマップアプリが開くリンク風、実際はモック）
    - 関連アーティストの名前リンク（タップで各アーティスト詳細へ）
    - 公式URL／チケットURLボタン
  - 下部: 「Googleカレンダーに追加」ボタン（primary）
- 挙動:
  - 「Googleカレンダーに追加」押下:
    - ボタンを即座に `display: none` にし、「Googleカレンダーを開いています...」インジケータを表示。
    - setTimeout(500〜800ms) 後、`window.open` で GoogleカレンダーURLモックを開く。
    - URLにはタイトル／日付／場所／詳細をクエリパラメータとしてエンコードして含める（あくまで表示用）。
  - ユーザーが戻るボタン連打などで画面遷移していた場合:
    - タイマーコールバック内で `state.currentView === "eventDetail"` かをチェックし、違う場合は何もせず終了する（ガード節）。

### 5.5 G4: 共有ページ
- 表示専用ビュー。URLに含まれる共有IDから ShareProfile を取得して表示。
- コンポーネント:
  - ヘッダー: 「シェアビュー / Shared view」、オーナー名のみ表示。
  - フォロー中アーティスト一覧（カード）
  - 「今後のイベント」リスト
- 非表示事項（明示的に仕様に含める）:
  - `goingFlag`, `ticketStatus`, `userNotesPrivate` はこの画面では一切使用しない。

### 5.6 G5: 設定
- コンポーネント:
  - 共有ページ公開トグル（ON/OFF）
  - 共有URL表示フィールド＋「コピー」ボタン
  - 「サンプルデータに戻す」ボタン
  - ログアウトボタン
- 挙動:
  - トグルON時:
    - ShareProfile がなければ新規作成して保存。
    - 共有URLを生成し、画面に表示。
  - トグルOFF時:
    - `isPublic` を false に更新。
  - 「サンプルデータに戻す」:
    - localStorage の対象キーを全て削除し、サンプルデータを再書き込み。
    - トースト「サンプルデータにリセットしました」を表示。

## 6. 状態遷移とバグ防止設計

### 6.1 画面遷移条件

| 遷移 | 前提条件 | 正常時動作 | 異常時の動作 |
|------|----------|------------|--------------|
| G0 → G1 | ユーザー選択完了（`currentUserId` が非null） | ホーム（G1）を表示 | `currentUserId` がnullならG0に留まり、エラートーストを表示 |
| G1 → G2 | `currentUserId` 非null ＆ 対象アーティストIDが存在 | `currentArtistId` をセットし、G2を表示 | アーティストIDが不正ならG1に戻り、「アーティストが見つかりません」トースト |
| G2 → G3 | `currentArtistId` 非null ＆ 対象イベントIDが存在 | `currentEventId` をセットし、G3を表示 | イベントID不正時はG2に戻り、「イベントが見つかりません」トースト |
| 任意 → G4 | 共有URLからアクセス、対応するShareProfileが存在し`isPublic`=true | G4を表示 | ShareProfileが無い or `isPublic`=false の場合は簡易な404画面を表示 |
| 任意 → G5 | `currentUserId` 非null | 設定画面を表示 | ログインしていない場合はG0にリダイレクト |

### 6.2 ボタン表示条件
- 「Googleカレンダーに追加」ボタン:
  - 表示条件: `currentView === "eventDetail"` かつ `currentEventId` に紐づく Event が存在し、`isLoading === false`。
  - 押下後: 即座に `display: none` にし、ローディング表示に切り替える。
- 「共有ページを公開」トグル:
  - 表示条件: `currentUserId` 非null。
- 「サンプルデータに戻す」ボタン:
  - 常に表示してよいが、押下後は一時的に無効化してトースト表示後に再度有効化。

### 6.3 表示／非表示実装ルール
- 画面／モーダル／ボタンの表示切替には、必ず `display: none` / `display: block` を使用する。
- `opacity: 0` や `pointer-events: none` のみで隠すことは禁止（見えないボタンが押せる状態を避ける）。

### 6.4 タイマーと非同期モック
- ログイン処理やGoogleカレンダー起動など、非同期っぽく見せたい箇所は `setTimeout` を使用。
- 各タイマーコールバックの先頭で、期待する状態かどうかを判定するガード節を必ず入れる:
  - 例: `if (state.currentView !== "eventDetail") return;`

### 6.5 ボタン要件
- すべての `<button>` 相当要素は、仕様として `type="button"` を付与することを明記。
- 送信フォームは存在しないため、`type="submit"` は使用しない。

## 7. サンプルデータ設計

### 7.1 アーティスト
- 最低3組のサンプル:
  - 日本語名のみ
  - 日本語＋英語名（カタカナ／英字混在）
  - 非常に長いアーティスト名（折り返し確認用）

### 7.2 イベント
- 各アーティストに対して3〜5件程度:
  - 平日夜開演のライブ
  - 週末昼のインストアイベント
  - 終日扱いの展示／ポップアップ
  - 同日・別会場のイベント（被り確認用）
  - 住所が長い会場（ビル名・階数入り）

### 7.3 リリース
- 各アーティストに対して2〜4件:
  - アルバム／EP／シングルが混在
  - 古いリリース（10年以上前）と新しいリリースを含める
  - タイトルが長いもの、日本語・英語・記号混在のもの

## 8. 技術メモ（プロトタイプ実装方針）

- 外部通信は行わず、すべてブラウザ内で完結させる。
- 認証:
  - G0で「Googleでログイン」ボタン→ユーザー選択→`currentUserId` セットという流れのみ。
  - 実際のGoogle OAuthは使用せず、UI上で連想しやすい見た目だけ再現する。
- Googleカレンダー連携:
  - 実際には `'https://calendar.google.com/calendar/render?...'` 形式のURL文字列を組み立てて `window.open` するだけ。
  - オフライン動作やCSPの関係で開けない場合もあるが、プロトタイプとしては「URLを生成しようとしている」ことが確認できればよい。
- 永続化:
  - localStorageにまとめて保存・読み出しするヘルパー関数を想定。
  - 「サンプルデータに戻す」操作で、全キーを削除してから初期データを再投入する。
- エラーハンドリング:
  - すべてのユーザー操作に対して、何らかの視覚的フィードバック（トースト、バナー等）を必ず表示する。
  - 予期しない状態（null参照など）は、「トップに戻す＋エラートースト」という安全な退避パターンを採用する。
